# 阶段 1 新手向知识点总结

> 面向：**第一次做「对话型智能体」**的同学。  
> 对应本项目：**`USE_RAG=false` 时的纯 LLM 多轮对话**（入口 `main.py` + `agent.py` + `config.py` + `prompts.py`）。

---

## 1. 智能体 / 对话应用，本质上在做什么？

| 概念 | 是什么 | 做什么用 | 为什么要用 |
|------|--------|----------|------------|
| **大语言模型（LLM）** | 能读自然语言、按概率生成下文的模型 | 把「用户说的话」变成「助理回复」 | 核心能力：理解意图、组织语言、结合提示词里的规则作答 |
| **对话应用** | 在你的程序里**多次调用**模型，并带上**历史** | 像聊天一样连续问 | 单轮只能回答当前句；多轮才能「接着聊」 |
| **阶段 1 在本项目里** | 不调知识库、不调外部工具 | 先跑通 **API + 多轮 + 人设** | 最简单可运行闭环，后面再加 RAG、工具、MCP 不会乱 |

---

## 2. Python 与项目结构（你要会看什么）

| 知识点 | 是什么 | 做什么用 | 为什么要用 |
|--------|--------|----------|------------|
| **`if __name__ == "__main__"`** | 判断「当前文件是被直接运行还是被 import」 | 直接运行才执行 `main()` | 同一文件既可当脚本，又可被别的模块引用 |
| **包与 `__init__.py`** | 目录被当成 Python **包** | 支持 `import src.xxx` | 工具链（Poetry、`-m` 运行）需要规范的包结构 |
| **`-m src.main`** | 以**模块方式**运行 | 从项目根解析包路径 | 避免「当前目录不对就 import 失败」 |
| **虚拟环境 + Poetry** | 依赖装在**独立环境**里 | `poetry install` / `poetry run` | 不同项目依赖版本互不污染、可复现 |

## 3. 配置与安全：`.env` 与 `config.py`

| 知识点 | 是什么 | 做什么用 | 为什么要用 |
|--------|--------|----------|------------|
| **环境变量** | 操作系统或进程里的「键值配置」 | 放 API Key、模型名等 | **不写进代码仓库**，换环境只改配置 |
| **`.env` + `python-dotenv`** | 本地文本里一行一个 `KEY=value`，启动时加载进环境 | 开发机方便改 Key | 与「12-factor」配置理念一致；`.gitignore` 忽略 `.env` 防泄露 |
| **`.env.example`** | **不含真实密钥**的模板 | 告诉同事要配哪些变量 | 仓库可提交，新人复制成 `.env` 即可 |
| **集中配置模块 `config.py`** | 统一读 env、做校验（如 Base URL 必须带 `https://`） | 业务代码只 `from src.config import ...` | **单一真相来源**，改接口地址只动一处 |

---

## 4. 调用模型：OpenAI 兼容接口

| 知识点 | 是什么 | 做什么用 | 为什么要用 |
|--------|--------|----------|------------|
| **API Key** | 平台发给你的凭证 | 请求时证明身份与计费 | 没有 Key 无法调用云端模型 |
| **`base_url`（如百炼）** | HTTP 请求的**根地址** | 把请求发到对应平台网关 | 官方 OpenAI、国内百炼、自建网关地址不同，但协议可兼容 |
| **超时 `timeout`** | 等响应的最长时间 | 弱网时避免无限卡住 | 用户体验与排错更清晰 |
| **OpenAI 兼容** | 请求/响应形状接近 OpenAI Chat API | 同一套客户端换 `base_url` 就能换后端 | 减少换厂商时重写一整层代码 |

## 5. LangChain 在本阶段用到的部分

| 知识点 | 是什么 | 做什么用 | 为什么要用 |
|--------|--------|----------|------------|
| **LangChain** | 围绕 LLM 应用的**组件库**（提示、链、记忆、工具、RAG 等） | 用现成抽象拼业务 | 少写胶水代码，后续扩展（工具、Agent）路径清晰 |
| **`ChatOpenAI`** | LangChain 封装的「聊天模型」客户端 | `invoke` / `generate` 发消息给 API | 与 LangChain 其它组件（Prompt、链）格式统一 |
| **`ChatPromptTemplate`** | **带槽位**的多条消息模板（system / human / …） | 把变量填进固定结构 | 人设、历史、当前问题**结构稳定**，减少字符串拼接错误 |
| **`MessagesPlaceholder("history")`** | 模板里预留「多轮消息」的位置 | 每轮把历史列表塞进去 | **多轮对话**的标准写法 |
| **LCEL 管道 `prompt \| llm`** | 竖线表示「前一个的输出给后一个」 | 一行定义「先渲染提示，再调模型」 | 可读、可组合，与阶段 2 的 `Lambda \| prompt \| llm` 同一套思路 |

## 6. System Prompt 与人设

| 知识点 | 是什么 | 做什么用 | 为什么要用 |
|--------|--------|----------|------------|
| **System 消息** | 对话里角色为「系统」的那条说明 | 定义助手身份、边界、风格 | 模型会**优先**遵守（相对 user 里的一句指令更稳） |
| **健康助理人设** | 本项目 `prompts.py` 里的 `SYSTEM_PROMPT` | 强调不诊断、不处方、建议就医 | **合规与产品边界**；医疗场景尤其重要 |
| **与代码分离** | Prompt 单独文件 | 改话术不改逻辑 | 产品/运营可协作迭代，减少动核心代码 |

---

## 7. 多轮对话：`history` 在本项目里怎么工作

| 知识点 | 是什么 | 做什么用 | 为什么要用 |
|--------|--------|----------|------------|
| **`history` 列表** | `main.py` 里存的 `(用户句, 助理句)` 元组列表 | 记住之前说过什么 | 模型本身**无状态**；状态由你的程序保存 |
| **`HumanMessage` / `AIMessage`** | LangChain 对「用户/助手」消息的封装 | 交给 `ChatPromptTemplate` | API 需要**带角色**的消息序列，不能直接塞一大段字符串 |
| **每轮 `invoke`** | 把「当前输入 + 展开后的历史」送进链 | 生成本轮回复 | 简单可靠；复杂场景以后可换「摘要记忆」「向量记忆」等 |

---

## 8. 入口与交互：`main.py` 在干什么

| 知识点 | 是什么 | 做什么用 | 为什么要用 |
|--------|--------|----------|------------|
| **CLI（命令行界面）** | `input` / `print` 与用户交互 | 最快验证逻辑 | 不接前端也能完整调试 |
| **输入校验** | 空行、`exit`/`quit` | 控制循环与退出 | 体验与资源占用更合理 |
| **异常分类处理** | 超时单独提示，其它错误 `raise` | 用户看得懂；开发仍能看到栈 | 上线产品与本地调试兼顾 |

## 9. 和阶段 2、3 的衔接（建立预期）

| 阶段 | 在阶段 1 基础上多什么 | 为什么后面要加 |
|------|------------------------|----------------|
| **阶段 2 RAG** | 检索本地文档，把片段塞进 Prompt | 回答**有据可查**，减少胡编（医疗科普尤其需要） |
| **阶段 3 ReAct + 工具** | 模型**决定何时**调用「查知识 / 查时间 / …」 | 更灵活；问题类型不同时走不同能力 |
| **MCP** | 用标准协议挂日历、文件等外部能力 | 扩展性强，工具可复用 |
